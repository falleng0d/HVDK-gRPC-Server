# GRPCRemote Windows Service Installer — Implementation Plan

## Overview

Two pieces of work are required:

1. **Modify `GRPCRemote` to run as a native Windows Service** — using
   `UseWindowsService()` from `Microsoft.Extensions.Hosting.WindowsServices`.
2. **Create a `GRPCRemoteInstaller` WiX 4 MSI project** — installs the
   binaries, registers the service with SCM, configures auto-start and
   failure-restart recovery.

The final deliverable is a single `.msi` file that installs GRPCRemote as a
Windows Service that:
- Starts automatically with Windows
- Passes `--urls http://0.0.0.0:9036` to the server at startup
- Restarts automatically if the process dies (via SCM failure actions)
- Can be uninstalled cleanly via Add/Remove Programs

---

## Part 1 — Modify GRPCRemote to support Windows Service mode

### 1.1 Add NuGet package

```
Microsoft.Extensions.Hosting.WindowsServices
```

Add to `GRPCRemote/GRPCRemote.csproj`:

```xml
<PackageReference Include="Microsoft.Extensions.Hosting.WindowsServices" Version="8.*" />
```

### 1.2 Call `UseWindowsService()` in `Program.cs`

Insert before `builder.Build()`:

```csharp
builder.Host.UseWindowsService(options =>
{
    options.ServiceName = "GRPCRemote";
});
```

**What this does:**
- When the process is started by SCM (Windows Service Control Manager), it
  hooks into the SCM lifecycle: Start, Stop, Pause, Resume.
- When started from a console (development/debugging), it behaves identically
  to the current console-app mode — no behavioural change for interactive use.
- Automatically sets `ContentRootPath = AppContext.BaseDirectory`, fixing a
  known issue where services run with `System32` as their working directory
  and fail to find `appsettings.json` / `grpc-remote.config.json`.
- Integrates with the Windows Event Log: log output from
  `ILogger` is forwarded there automatically.

### 1.3 File logging (recommended)

When running as a service, stdout is not visible. Add a file sink via
`Serilog` or the built-in `Microsoft.Extensions.Logging` file provider so
there is a persistent log for diagnostics.

Recommended: Add `Serilog.Extensions.Hosting` + `Serilog.Sinks.File` and
write to `{AppContext.BaseDirectory}\logs\grpcremote-.log` with daily rolling.

### 1.4 No changes needed to `--urls` handling

`WebApplication.CreateBuilder(args)` already passes command-line args to the
Kestrel URL binding. The service registration (Part 2) sets `--urls
http://0.0.0.0:9036` in the `binPath`, so the URL is passed into the process
automatically when SCM starts it.

## Part 2 — Create `GRPCRemoteInstaller` WiX 4 MSI project

### 2.1 Prerequisites: install WiX toolchain

WiX 4 ships as a .NET global tool and NuGet SDK — no separate installer.

```powershell
dotnet tool install --global wix
```

This provides the `wix.exe` CLI and the MSBuild SDK.

### 2.2 New project: `GRPCRemoteInstaller/GRPCRemoteInstaller.wixproj`

WiX 4 uses `WixToolset.MSBuild` as the SDK. The project file:

```xml
<Project Sdk="WixToolset.MSBuild">
  <PropertyGroup>
    <ProductVersion>1.0.0</ProductVersion>
    <OutputName>GRPCRemoteInstaller</OutputName>
    <Platform>x64</Platform>
  </PropertyGroup>

  <!-- Harvest published files from GRPCRemote publish output -->
  <PropertyGroup>
    <PublishDir>$(MSBuildThisFileDirectory)..\GRPCRemote\bin\Publish\</PublishDir>
  </PropertyGroup>

  <ItemGroup>
    <PackageReference Include="WixToolset.Heat" Version="4.*" />
    <PackageReference Include="WixToolset.Util.wixext" Version="4.*" />
  </ItemGroup>

  <ItemGroup>
    <Compile Include="Product.wxs" />
    <Compile Include="Harvest.wxs" /> <!-- auto-generated by heat -->
  </ItemGroup>
</Project>
```

### 2.3 Publish GRPCRemote before building the MSI

The MSI must embed the GRPCRemote binaries. Two options:

| Mode | Command | Output size | .NET runtime needed on target |
|------|---------|-------------|-------------------------------|
| **Framework-dependent** (recommended) | `dotnet publish -c Release -r win-x64 --no-self-contained` | ~5 MB | Yes — .NET 8.0 ASP.NET Core Runtime |
| Self-contained | `dotnet publish -c Release -r win-x64 --self-contained true` | ~120 MB | No |

**Recommendation:** Framework-dependent. The machine already has the .NET 8
SDK installed (required to build). The ASP.NET Core Runtime redistributable
can be checked/installed by the MSI if needed (see §2.6).

Publish command:

```powershell
dotnet publish GRPCRemote\GRPCRemote.csproj `
    -c Release -r win-x64 --no-self-contained `
    -o GRPCRemote\bin\Publish
```

### 2.4 File harvesting

WiX 4 includes `wix.exe harvest dir` to auto-generate a component group from
the publish output. Run once to generate `Harvest.wxs`, then commit it:

```powershell
wix harvest dir GRPCRemote\bin\Publish `
    -componentgroupname PublishedFiles `
    -directoryrefid INSTALLFOLDER `
    -dr INSTALLFOLDER `
    -srd `         # suppress root directory component
    -out GRPCRemoteInstaller\Harvest.wxs
```

Re-run this command whenever the published output changes (e.g. added
dependencies after a NuGet update). The `Harvest.wxs` file should be
committed to source control.

### 2.5 Main WiX source: `GRPCRemoteInstaller/Product.wxs`

```xml
<?xml version="1.0" encoding="UTF-8"?>
<Wix xmlns="http://wixtoolset.org/schemas/v4/wxs"
     xmlns:util="http://wixtoolset.org/schemas/v4/wxs/util">

  <Package
    Name="HVDK GRPCRemote"
    Manufacturer="Tetherscript"
    Version="$(var.ProductVersion)"
    UpgradeCode="PUT-A-NEW-GUID-HERE"
    Language="1033"
    Codepage="1252"
    InstallerVersion="500">

    <!-- Prevent downgrade -->
    <MajorUpgrade DowngradeErrorMessage="A newer version is already installed." />

    <!-- Installation directory -->
    <StandardDirectory Id="ProgramFiles64Folder">
      <Directory Id="INSTALLFOLDER" Name="GRPCRemote" />
    </StandardDirectory>

    <!-- Feature: the service -->
    <Feature Id="ProductFeature" Title="GRPCRemote Service" Level="1">
      <ComponentGroupRef Id="PublishedFiles" />   <!-- from Harvest.wxs -->
      <ComponentRef Id="ServiceComponent" />
    </Feature>

    <!-- Service registration component -->
    <Component Id="ServiceComponent" Directory="INSTALLFOLDER" Guid="PUT-A-NEW-GUID-HERE">

      <!-- Register the Windows Service -->
      <ServiceInstall
        Id="GRPCRemoteServiceInstall"
        Name="GRPCRemote"
        DisplayName="HVDK gRPC Remote Input Server"
        Description="Exposes HVDK virtual HID driver input over gRPC on port 9036."
        Type="ownProcess"
        Start="auto"
        ErrorControl="normal"
        Account="LocalSystem"
        Arguments='--urls "http://0.0.0.0:9036"' />

      <!-- Start on install, stop on uninstall, delete on uninstall -->
      <ServiceControl
        Id="GRPCRemoteServiceControl"
        Name="GRPCRemote"
        Start="install"
        Stop="both"
        Remove="uninstall"
        Wait="yes" />

      <!-- Failure actions: restart with increasing delay, reset counter daily -->
      <!-- Requires MSI 5.0 (Windows 10 / Server 2016+) — safe for .NET 8 targets -->
      <ServiceConfigFailureActions
        ServiceName="GRPCRemote"
        OnInstall="yes"
        OnReinstall="yes"
        ResetPeriod="86400">
        <Failure Action="restart" Delay="30000"  />  <!-- 1st failure: restart after 30s -->
        <Failure Action="restart" Delay="60000"  />  <!-- 2nd failure: restart after 60s -->
        <Failure Action="restart" Delay="120000" />  <!-- subsequent: restart after 120s -->
      </ServiceConfigFailureActions>

    </Component>

  </Package>
</Wix>
```

**Key attribute notes:**

| Attribute | Value | Reason |
|-----------|-------|--------|
| `Start="auto"` | Automatic | Service starts with Windows |
| `Account="LocalSystem"` | LocalSystem | Needed for HID driver access; can be tightened later |
| `Arguments='--urls "http://0.0.0.0:9036"'` | URL binding | Passed to the exe in the SCM binary path |
| `Stop="both"` | Stop on install and uninstall | Avoids stale process during upgrade |
| `ResetPeriod="86400"` | 1 day in seconds | Failure counter resets after 24h |

### 2.6 Optional: .NET runtime prerequisite check

If the target machine may not have .NET 8 ASP.NET Core Runtime installed, add
a bootstrapper bundle (`Bundle.wxs`) using `WixToolset.Netfx.wixext` or a
`util:RegistrySearch` condition:

```xml
<!-- Fail install if ASP.NET Core 8.0 Runtime is missing -->
<util:RegistrySearch
  Id="CheckDotNet8"
  Root="HKLM"
  Key="SOFTWARE\dotnet\Setup\InstalledVersions\x64\sharedhost"
  Name="Version"
  Result="value"
  Variable="DOTNET8_INSTALLED" />

<Condition Message="This application requires ASP.NET Core Runtime 8.0 or later.">
  <![CDATA[Installed OR DOTNET8_INSTALLED]]>
</Condition>
```

A simpler alternative: bundle the .NET 8 runtime installer inside a WiX
`Bundle` bootstrapper project (`GRPCRemoteBundle.wixproj`), but this is only
worthwhile if distribution to machines without .NET is expected.

### 2.7 Add project to Utils.sln

```
dotnet sln Utils.sln add GRPCRemoteInstaller\GRPCRemoteInstaller.wixproj
```

## Part 3 — Build process

Full build sequence:

```powershell
# 1. Publish GRPCRemote (framework-dependent, win-x64)
dotnet publish GRPCRemote\GRPCRemote.csproj `
    -c Release -r win-x64 --no-self-contained `
    -o GRPCRemote\bin\Publish

# 2. (First time, or when dependencies change) Re-harvest published files
wix harvest dir GRPCRemote\bin\Publish `
    -componentgroupname PublishedFiles `
    -directoryrefid INSTALLFOLDER -dr INSTALLFOLDER -srd `
    -out GRPCRemoteInstaller\Harvest.wxs

# 3. Build MSI
dotnet build GRPCRemoteInstaller\GRPCRemoteInstaller.wixproj -c Release
# Output: GRPCRemoteInstaller\bin\Release\GRPCRemoteInstaller.msi
```

## Part 4 — Installing and managing the service

```powershell
# Install (runs as Administrator)
msiexec /i GRPCRemoteInstaller.msi

# Silent install
msiexec /i GRPCRemoteInstaller.msi /quiet

# Uninstall
msiexec /x GRPCRemoteInstaller.msi

# Manual service control after install
sc start GRPCRemote
sc stop  GRPCRemote
sc query GRPCRemote
```

The installer also appears in **Control Panel → Programs → Uninstall a
program** under "HVDK GRPCRemote".

## Decision points / open questions

| # | Question | Recommendation |
|---|----------|----------------|
| 1 | Self-contained vs framework-dependent publish? | Framework-dependent — .NET 8 is already a dev dependency |
| 2 | .NET runtime check in installer? | Add a registry condition; bundle bootstrapper only if needed |
| 3 | Service account: LocalSystem vs a dedicated account? | LocalSystem for now (HID driver requires it); revisit for hardening |
| 4 | Add Serilog file logging? | Yes — required for service diagnostics since no console is visible |
| 5 | Configurable port in installer? | Out of scope for now; `9036` is hardcoded in `Arguments` |
| 6 | WiX bundle bootstrapper (EXE wrapping the MSI)? | Not needed unless distributing to machines without .NET 8 |
